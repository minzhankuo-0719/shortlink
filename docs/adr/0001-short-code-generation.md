# ADR 0001 — 短碼產生方式：隨機 base62

## 決策

短碼用 `secrets.choice`（CSPRNG，密碼學安全的隨機數產生器）從 base62 字元集（0-9, a-z, A-Z）隨機取樣 7 個字元，寫入前查重複，若撞到已存在的碼就重抽（最多重試 5 次）。

實作見 [`apps/shortener/services.py`](../../apps/shortener/services.py) 的 `generate_short_code()`。

## 為什麼

短網址服務的短碼設計，常見三種做法，逐一比較：

1. **自增 ID 轉 base62**（如 1 -> "1"、2 -> "2"…）
   - 優點：簡單、不會碰撞。
   - 缺點：**可被列舉**。攻擊者依序造訪 `/1`, `/2`, `/3`…就能爬出全站所有短網址，等於洩漏「目前總共建立了多少條」與「別人建立了什麼」。對外服務不該讓使用量規模這麼容易被推算出來。
2. **URL hash**（對 `original_url` 做雜湊取前幾碼）
   - 優點：同一個 URL 永遠對應同一個短碼，天然去重。
   - 缺點：**會碰撞**（不同 URL hash 出同一個前綴），且**無法支援同一個原始網址被不同使用者各自建立自己的短碼**（這違反「每個使用者的短網址都是自己的資源」這個資料模型假設）。
3. **隨機 base62 + unique 檢查 + 碰撞重試**（採用）
   - 優點：不可列舉（隨機）、同一網址不同使用者可各自產生不同短碼、7 碼 base62 有 62⁷ ≈ 3.5 兆種組合，隨機碰撞機率極低。
   - 代價：每次建立都要查一次資料庫確認沒撞號（`short_code` 欄位已是 `unique=True`，這個查詢走索引，成本很小）；理論上仍有極小機率連續撞號，故設重試上限，超過視為異常直接拋例外（在 3.5 兆分之一機率下連撞 5 次幾乎不可能發生，真的發生代表系統另有問題，不該默默吞掉）。

## 替代方案被否決的理由

如上 1、2 點。
