# ADR 0002 — 重導用 302 而非 301

## 決策

`redirect_short_link` view 用 Django `redirect()` 預設的 **302（Found / 暫時重導）**，不用 **301（Moved Permanently / 永久重導）**。

## 為什麼

瀏覽器對待 301 和 302 的快取行為不同：

- **301**：瀏覽器認定「這個網址永遠都會跳到那裡」，會把這個對應關係**快取在本機**。使用者第一次點擊後，瀏覽器之後直接跳轉到目的網址，**不會再發請求到我們的伺服器**。
- **302**：瀏覽器認定「這次先這樣跳，下次可能不一樣」，**每次點擊都會重新發請求**到我們的伺服器，由伺服器決定要跳去哪裡。

本服務的核心需求之一是「看到自己縮網址的點擊成效」——如果用 301，使用者（或他朋友）只要點過一次，瀏覽器快取生效後，**後續點擊都不會再經過我們的伺服器，點擊數會被嚴重低估**，分析數據失真。302 確保每次點擊都真的打到伺服器，`Click` 才能被正確記錄。

## 代價與取捨

302 代表瀏覽器不會快取重導結果，每次點擊都會多一次到伺服器的請求往返。但對縮網址服務而言，**點擊數據的正確性比省這一次請求更重要**——這正是這個決策要對面試官講清楚的判斷依據。Stage 4 的 Redis cache-aside 是用來補償「每次都要查資料庫」的效能成本，而不是回頭改用 301（兩者解決的是不同問題：301 省的是瀏覽器到伺服器的網路往返，cache-aside 省的是伺服器內部查資料庫的延遲）。
