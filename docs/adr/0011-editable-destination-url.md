# ADR 0011 — 短網址可編輯目的地 URL

## 背景

編輯功能讓連結擁有者可以修改 `title` 與 `original_url`。`title` 是純標示文字,改它沒有任何
副作用。真正要決策的是 `original_url`——也就是重導的「目的地」——該不該開放修改。短網址的
核心語意是「一個穩定的指標」:`/abc123` 一旦發出去,別人可能已經分享、貼在文章或訊息裡。
讓目的地可變,等於可以「換掉所有持有此連結者最終會到的地方」。

## 決策

**開放編輯 `title` 與 `original_url` 兩者**,但 `short_code` 維持不變(編輯不換碼,持有舊
連結的人仍指向同一條 ShortLink)。介面用獨立編輯頁(`/links/<pk>/edit/`),view `edit_link`
以 `get_object_or_404(ShortLink, pk=pk, owner=request.user)` 做 owner-scope:只有擁有者能
編輯,他人或不存在一律 404(與 `delete_link` 同一防護模式)。表單沿用 `ShortLinkForm`,綁
`instance=link`——GET 預填現值、`form.save()` 更新同一筆,套用與建立相同的驗證。

## 為什麼技術上安全

改目的地最怕「快取還指著舊網址」。但失效已在 ADR 0010 用 `post_save` 訊號集中處理:
`form.save()` 觸發訊號 → 清掉該 `short_code` 的 Redis 快取 → 下一次重導讀到新值。**因為
失效寫在訊號而非建立流程,編輯不需補任何快取程式碼就正確**——這正是當初把失效集中在訊號
的回報。

## 代價與取捨

- **語意風險(bait-and-switch)**:目的地可變代表「先分享無害連結、之後改指惡意網站」在技術
  上可行。本服務接受這個風險,因為它是**個人、owner-scoped 的連結管理工具**(非公共代縮
  服務),且 Bitly 等真實產品的付費方案也提供改目的地。若日後轉為公共服務,應重新評估
  (例如限制改動頻率、保留稽核紀錄、或鎖定已大量點擊的連結)。
- **點擊統計會延續**:因為編輯是「更新同一筆 ShortLink」而非「重建」,既有的 `Click` 仍掛
  在這條連結上,點擊數會跨越這次改動繼續累計。這是刻意的——短碼被視為穩定的身分,統計跟著
  身分走。若想要「改了 URL 就歸零統計」,得改成重建一筆新連結(並換新短碼),但那就破壞了
  「同一個短碼」的承諾,故不採用。
- **未開放改 `short_code`**:換短碼等於讓舊連結直接失效(404),傷害更大,且會引入碰撞/唯一性
  處理。維持短碼不可變,讓編輯的範圍乾淨。

## 驗證

owner A 編輯自己的連結 → 302 導回列表、值已更新且短碼不變;owner B 對 A 的連結打編輯 URL
→ 404;改 `original_url` 後立刻造訪短碼 → 重導到新目的地(訊號已清快取)。
